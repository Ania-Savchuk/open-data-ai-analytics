# Звіт про виконання лабораторної роботи 1

---

### 1. Мета та завдання роботи
Метою проєкту є побудова повного циклу аналізу даних (Data Pipeline) з використанням мови Python та демонстрація навичок командної розробки через систему Git. 

**Основні завдання:**
* Налаштування професійного середовища розробки та структури репозиторію.
* Автоматизація збору даних із зовнішніх веб-ресурсів.
* Проведення багатофакторного аналізу якості даних та їх очищення.
* Статистична перевірка гіпотез щодо ринку ТЗ.
* Візуалізація результатів для формування бізнес-висновків.

---

### 2. Характеристика джерела та полів даних
Об'єктом дослідження є Реєстр зареєстрованих транспортних засобів МВС України.

#### Опис полів датасету (Data Dictionary):
| Назва поля | Опис характеристики | Тип у Pandas |
| :--- | :--- | :--- |
| **BRAND** | Марка автомобіля (уніфікована) | object (category) |
| **MODEL** | Модель автомобіля | object |
| **MAKE_YEAR** | Рік випуску (діапазон 1900-2026) | int64 |
| **FUEL** | Тип пального (Бензин, Дизель, Електро тощо) | object (category) |
| **CAPACITY** | Робочий об'єм двигуна (см³) | float64 |
| **OWN_WEIGHT** | Маса без навантаження (кг) | float64 |
| **TOTAL_WEIGHT** | Повна дозволена маса (кг) | float64 |
| **COLOR** | Колір кузова | object |
| **OPER_NAME** | Тип реєстраційної дії | object |

### 3. Хід виконання роботи

#### Етап 1. Ініціалізація та архітектура репозиторію
1.  **Ініціалізація Git:** Створено робочу директорію та проведено ініціалізацію репозиторію командою `git init`.
2.  **Створення структури:** Сформовано логічну структуру папок:
    * `data/` — для даних;
    * `notebooks/` — для Jupyter-ноутбуків;
    * `src/` — для Python-скриптів;
    * `reports/figures/` — для збережених графіків.
3.  **Налаштування .gitignore:** Додано виключення для великих CSV-файлів (`data/raw/`), кешу Python та віртуальних середовищ.
4.  **Документація:** Заповнено файл `README.md`, де вказано мету, джерело даних та три аналітичні гіпотези.

#### Етап 2. Розробка модуля завантаження (Гілка feature/data_load)
1.  Створено окрему гілку для розробки функціоналу завантаження.
2.  Розроблено скрипт `src/load_data.py`, який за допомогою бібліотеки `requests` завантажує архів і розпаковує його безпосередньо в пам'яті (`zipfile` + `io.BytesIO`).
3.  Виконано злиття (`merge`) гілки у `main` після успішного тестування скрипта.

#### Етап 3. Аналіз якості та дослідження (Feature-гілки)
Для розділення етапів розробки було створено дві окремі гілки:
* **feature/data_quality_analysis:** Створено ноутбук для аудиту даних. Виявлено 710 дублікатів та пропуски у полях `CAPACITY` та `FUEL`. Проведено валідацію років випуску.
* **feature/data_research:** Реалізовано очищення даних та розрахунки для перевірки гіпотез:
    * **Гіпотеза 1:** Підтверджено, що Топ-10 брендів займають **54.36%** ринку.
    * **Гіпотеза 2:** Визначено частку електромобілів — **4.85%**.
    * **Гіпотеза 3:** Розраховано медіанний вік автопарку — **14 років**.
* **Результат:** Обидві гілки злито в `main` через процедуру Pull Request з детальним описом внесених змін.

#### Етап 4. Моделювання merge-конфлікту
Для відпрацювання навичок вирішення конфліктів створено дві гілки, що одночасно змінювали одну й ту саму секцію в `README.md`. Після виникнення конфлікту під час злиття файл було відредаговано вручну: видалено службові маркери `<<<<<<<`, `=======` та `>>>>>>>`, обрано фінальний варіант тексту та зафіксовано зміни новим комітом.

#### Етап 5. Візуалізація та версіонування
1.  **Візуалізація:** У гілці `feature/visualization` створено ноутбук `visualization.ipynb`. За допомогою бібліотеки `Seaborn` побудовано графіки розподілу брендів, типів пального та гістограму віку авто.
2.  **Документування:** Додано файл `CHANGELOG.md` із хронологією розробки та описом ключових змін.
3.  **Реліз:** Створено тег версії `v0.1.0` для фіксації фінального стабільного стану проєкту.
