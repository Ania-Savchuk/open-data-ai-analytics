# Звіт про виконання лабораторної роботи 1

---

### 1. Мета та завдання роботи
Метою проєкту є побудова повного циклу аналізу даних (Data Pipeline) з використанням мови Python та демонстрація навичок командної розробки через систему Git. 

**Основні завдання:**
* Налаштування професійного середовища розробки та структури репозиторію.
* Автоматизація збору даних із зовнішніх веб-ресурсів.
* Проведення багатофакторного аналізу якості даних та їх очищення.
* Статистична перевірка гіпотез щодо ринку ТЗ.
* Візуалізація результатів для формування бізнес-висновків.

---

### 2. Характеристика джерела та полів даних
Об'єктом дослідження є Реєстр зареєстрованих транспортних засобів МВС України.

#### Опис полів датасету (Data Dictionary):
| Назва поля | Опис характеристики | Тип у Pandas |
| :--- | :--- | :--- |
| **BRAND** | Марка автомобіля (уніфікована) | object (category) |
| **MODEL** | Модель автомобіля | object |
| **MAKE_YEAR** | Рік випуску (діапазон 1900-2026) | int64 |
| **FUEL** | Тип пального (Бензин, Дизель, Електро тощо) | object (category) |
| **CAPACITY** | Робочий об'єм двигуна (см³) | float64 |
| **OWN_WEIGHT** | Маса без навантаження (кг) | float64 |
| **TOTAL_WEIGHT** | Повна дозволена маса (кг) | float64 |
| **COLOR** | Колір кузова | object |
| **OPER_NAME** | Тип реєстраційної дії | object |

### 3. Детальний хід виконання роботи

#### Етап 1. Ініціалізація та Git-архітектура
1.  Створено локальний та віддалений репозиторій `open-data-ai-analytics`.
2.  Розгорнуто стандартну структуру папок: `data/`, `notebooks/`, `src/`, `reports/`.
3.  Створено файл `.gitignore`, куди додано виключення для великих CSV-файлів (`data/raw/`), кешу Python та локальних конфігурацій середовища (`.venv`, `.env`).
4.  Сформовано файл `README.md` із описом мети та трьох дослідницьких гіпотез.

#### Етап 2. Автоматизація завантаження (Гілка `feature/data_load`)
1.  Розроблено модуль `src/load_data.py`. 
2.  Реалізовано логіку завантаження через бібліотеку `requests` з імітацією заголовків браузера (`User-Agent`).
3.  Для оптимізації швидкодії використано бібліотеку `zipfile` та `io.BytesIO`, що дозволило розпакувати ZIP-архів у оперативній пам'яті без проміжних записів на диск.

#### Етап 3. Аудит якості (Гілка `feature/data_quality_analysis`)
1.  Створено ноутбук `data_quality_analysis.ipynb`.
2.  Виявлено технічний «шум» у розмірі **710 дублікатів**.
3.  Проведено аналіз пропущених значень (Missing Values): встановлено критичні пропуски в характеристиках двигунів (8.2%).
4.  Перевірено цілісність даних: підтверджено відсутність аномальних значень у роках випуску (валідація меж 1900-2026).

#### Етап 4. Аналіз та очищення (Гілка `feature/data_research`)
1.  Виконано Data Cleaning: видалення дублікатів та заміна неінформативних маркерів у полі пального (`.`, `ВІДСУТНЄ`).
2.  Проведено статистичну перевірку гіпотез:
    * **Гіпотеза 1:** Виявлено концентрацію ринку (Топ-10 брендів займають **54.36%**).
    * **Гіпотеза 2:** Розраховано частку електромобілів (**4.85%**).
    * **Гіпотеза 3:** Визначено медіанний вік автопарку (**14 років**).

#### Етап 5. Візуалізація та конфлікти (Гілка `feature/visualization`)
1.  Підготовлено фінальні графіки в `visualization.ipynb` з використанням бібліотеки `Seaborn`.
2.  Штучно змодельовано **Merge Conflict** у файлі `README.md` між двома гілками для відпрацювання навичок ручного вирішення конфліктів через стандартні маркери Git (`<<<<<<< HEAD`).

#### Етап 6. Фіналізація та реліз
1.  Створено `CHANGELOG.md` за принципами Semantic Versioning.
2.  Зафіксовано стан проєкту через створення тегу версії: `git tag -a v0.1.0 -m "Release version 0.1.0"`.
